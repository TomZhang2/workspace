## 一、核心研究发现

经过对 postgres_fdw / Trino / Spark / Calcite / Iceberg 的深度研究，提炼出 **10 条决定性洞察**，每条直接驱动一个设计决策：

| #   | 洞察                                                 | 来源                                | 驱动的设计决策                                                |
| --- | -------------------------------------------------- | --------------------------------- | ------------------------------------------------------ |
| 1   | **不贪心切割，而是枚举候选路径让 cost 选**                         | postgres_fdw + Calcite            | 下推分析器构建多条路径，规划器按 cost 选最优                              |
| 2   | **可行性判定与执行分离**（`*_ok` 先检查，deparse 只对选中路径执行）        | postgres_fdw                      | `isPushable()` 纯检查 + `applyXxx()` 仅对选中路径执行             |
| 3   | **EXACT vs CONSERVATIVE 两种模式不可混淆，且需逐 conjunct 区分** | Iceberg 研究 + v2 修复                | `ConjunctPushdown` 逐 conjunct 模式 + 残余不变量校验器            |
| 4   | **Inclusive/Strict 投影对偶**                          | Iceberg                           | 数据跳过源用 Inclusive（可能匹配→读），SQL 源用 Strict（必定匹配→跳过评估）      |
| 5   | **残余是一等公民，每个算子都返回残余**                              | Trino + postgres_fdw              | 所有 `applyXxx` 返回 `(pushedHandle, residual)`，引擎据此补全     |
| 6   | **静态+动态可组合**（`CompletableFuture` + 不可变 `Snapshot`） | Trino DynamicFilter               | 同一扫描同时享受规划期谓词和运行期谓词                                    |
| 7   | **guarantee flag 与下推成功解耦，且需 collation 条件**         | Spark + postgres_fdw + v2 修复      | guarantee flag 独立 + 携带 collation 证明 + 默认安全(false)      |
| 8   | **谓词必须是表达式 IR，不能仅用 TupleDomain**                   | Trino ConnectorExpression + v2 修复 | `ConnectorExpression` IR 替换 `TupleDomain`，支持函数/算术/cast |
| 9   | **部分聚合需声明中间状态与合并函数**                               | Spark/Trino merge() + v2 修复       | `IntermediateAggregate` 暴露中间类型 + `MergeFunction`       |
| 10  | **成本模型需要统计来源，否则退化为启发式**                            | postgres_fdw stats + v2 修复        | `StatisticsProvider` + `FallbackEstimator` 回退估计器       |

### v2 相对 v1 的核心改进

v1 是一份优秀的**研究综述**，但不是一份**可实施规范**。Oracle 架构评审发现 5 个关键正确性/完整性问题，v2 逐一修复：

| v1 问题                                          | v2 修复                                                                               |
| ---------------------------------------------- | ----------------------------------------------------------------------------------- |
| `TupleDomain` 只能表达列域谓词，无法推 `UPPER(name)='BOB'` | 引入 `ConnectorExpression` IR，支持函数/算术/cast                                            |
| 单一 `PushdownMode` 无法表达混合语义(Hive)               | 改为逐 conjunct 模式 `List<ConjunctPushdown>`                                            |
| `applyFilter` 合并检查与执行，违背自身原则                   | 拆为 `isPushable`（纯检查）+ `applyXxx`（仅选中路径执行）                                           |
| 部分聚合无合并契约，`remainingHaving` 类型错误               | `IntermediateAggregate` + `MergeFunction`；`remainingHaving` 改 `ConnectorExpression` |
| 成本模型无统计来源，"成本驱动"是虚构                            | `StatisticsProvider` + `FallbackEstimator` 回退估计器                                    |
| guarantee flag 被盲目信任                           | 携带 collation 证明 + 默认安全(false) + 一致性测试                                               |
| 无错误回退/事务/RLS/子查询/测试套件                          | 新增 6 个章节覆盖                                                                          |
| 路径爆炸未解决                                        | 加 memo + 剪枝 + 候选上限                                                                  |

## 三、统一下推契约（核心 SPI）

### 3.1 ConnectorExpression IR（v2 新增，修复 Top5 #1）

v1 使用 `TupleDomain<ColumnHandle>` 表达谓词，仅支持列域范围（`col > 5`、`col IN (...)`），无法表达函数调用（`UPPER(name) = 'BOB'`）、算术（`a + b > 10`）、cast（`CAST(x AS INT)`）。v2 引入 `ConnectorExpression` IR（参考 Trino ConnectorExpression）：

```java
/**
 * 下推谓词/投影/HAVING/join 条件的统一表达式 IR。
 * 取代 v1 的 TupleDomain<ColumnHandle>，支持任意表达式。
 */
public sealed interface ConnectorExpression
    permits Variable, Constant, Call, Comparison, Logical, Cast, Special, TupleDomain {}

/** 列引用 */
public record Variable(ColumnHandle column, Type type) implements ConnectorExpression {}

/** 字面量 */
public record Constant(Object value, Type type) implements ConnectorExpression {}

/** 函数调用：UPPER(name), date_trunc('month', ts), abs(x)
 *  注：volatility 通过 f.function().volatility() 访问（FunctionSignature 暴露 volatility 字段）
 */
public record Call(FunctionSignature function, List<ConnectorExpression> args, Type type) 
    implements ConnectorExpression {}

/**
 * 函数签名：携带函数名、参数类型、返回类型、volatility 分类。
 * volatility 驱动可运性判定（IMMUTABLE 可运，VOLATILE 不可运，STABLE 需快照钉入）。
 */
public record FunctionSignature(
    String name,                          // 函数名（如 "upper", "abs", "+"）
    List<Type> parameterTypes,
    Type returnType,
    FunctionVolatility volatility         // 从源 catalog 获取
) {}

/** 比较操作：=, <>, <, >, <=, >= */
public record Comparison(Operator op, ConnectorExpression left, ConnectorExpression right) 
    implements ConnectorExpression {}

/** 逻辑组合：AND, OR, NOT */
public record Logical(LogicalOperator op, List<ConnectorExpression> terms) 
    implements ConnectorExpression {}

/** 类型转换 */
public record Cast(ConnectorExpression expr, Type targetType) implements ConnectorExpression {}

/** 特殊谓词：IS NULL, IS NOT NULL, IN, BETWEEN, LIKE */
public record Special(SpecialKind kind, ConnectorExpression expr, List<ConnectorExpression> args) 
    implements ConnectorExpression {}

/**
 * TupleDomain 作为快速路径子集保留。
 * 纯范围源（Iceberg 统计）可仅产出 TupleDomain；混合源可部分用 TupleDomain 部分用 Call。
 */
public record TupleDomain(Map<ColumnHandle, Domain> domains) implements ConnectorExpression {

    /** 尝试将 ConnectorExpression 降级为 TupleDomain（用于快速路径优化） */
    public static Optional<TupleDomain> asTupleDomain(ConnectorExpression expr) {
        // 仅当表达式是纯列域比较的 AND 组合时可降级
    }
}
```

> **关键**：所有 `applyXxx` 方法的谓词参数、所有结果类型的残余字段，均使用 `ConnectorExpression` 而非 `TupleDomain`。`TupleDomain` 仅作为快速路径子集保留。

### 3.2 三种执行模式（逐 conjunct）

```java
public enum PushdownMode {
    EXACT,          // 源完整执行此 conjunct，结果可信（JDBC SQL 下推）
                    // 此 conjunct 的残余可为空

    CONSERVATIVE,   // 引擎用源元数据本地跳过，只证伪不证真（Iceberg/Delta/Parquet）
                    // 此 conjunct 的残余必须 == 原始 conjunct

    IN_MEMORY       // 全扫后引擎内存过滤
                    // 此 conjunct 的残余 == 原始 conjunct
}
```

> **v2 关键变更**：模式是**逐 conjunct** 的，而非整个结果一个模式。一个 Hive 查询可能同时有 EXACT conjunct（WHERE 子句下推到 HiveQL）和 CONSERVATIVE conjunct（ORC 统计跳过），v1 的单一模式枚举无法表达此混合语义。

### 3.3 残余不变量校验器（v2 新增，修复 Top5 #2）

```java
/**
 * 框架级残余不变量校验器（debug/test 构建启用）。
 * 确保 Connector 返回的残余与模式语义一致，防止静默错误结果。
 */
public class ResidualInvariantValidator {

    /**
     * 校验谓词下推结果的残余不变量。
     * 
     * 不变量：
     * - EXACT: originalConjunct ≡ pushed ∧ residual（逻辑等价）
     * - CONSERVATIVE: residual == originalConjunct（残余必须保留完整原始谓词）
     * - IN_MEMORY: residual == originalConjunct（无任何下推）
     */
    public void validateFilter(ConnectorExpression original, List<ConjunctPushdown> result) {
        // 将 original 拆分为 conjunct 列表
        List<ConnectorExpression> originalConjuncts = splitConjuncts(original);

        // 每个结果 conjunct 必须对应一个原始 conjunct
        for (ConjunctPushdown cp : result) {
            switch (cp.mode()) {
                case EXACT -> {
                    // 校验: pushed ∧ residual ≡ original
                    assertLogicalEquivalent(
                        and(cp.pushedExpression(), cp.residualExpression()),
                        cp.originalConjunct()
                    );
                }
                case CONSERVATIVE -> {
                    // 校验: residual == original（完整保留）
                    assertEquals(cp.residualExpression(), cp.originalConjunct());
                }
                case IN_MEMORY -> {
                    // 校验: residual == original, pushed 为空
                    assertEquals(cp.residualExpression(), cp.originalConjunct());
                    assertTrue(cp.pushedExpression().isEmpty());
                }
            }
        }
    }

    /**
     * 聚合残余校验：remainingHaving 必须是原始 HAVING 的子集
     */
    public void validateAggregate(ConnectorExpression originalHaving, 
                                   ConnectorExpression remainingHaving) {
        // remainingHaving 必须逻辑蕴含于 originalHaving
        assertImplies(remainingHaving, originalHaving);
    }
}
```


**铁律**：

1. `isXxxPushable` 必须无副作用、可缓存——用于路径构建阶段批量检查。
2. `applyXxx` 仅对规划器选中的路径调用——避免为未选中路径生成 SQL。
3. `applyXxx` 返回 `Optional.empty()` = 无效果（防死循环）。
4. 所有谓词参数使用 `ConnectorExpression`，不用 `TupleDomain`。

### 3.5 结果类型（v2 修订：逐 conjunct 模式 + 中间状态 + collation 证明）

```java
// ====== 谓词下推结果（v2.1: 全部改为 interface，对齐 §17.2 可扩展性）======

/**
 * 单个 conjunct 的下推结果。
 * v2 关键变更：每个 conjunct 独立声明其 PushdownMode。
 * v2.1: 改为 interface（record 不可扩展，违背 SPI 演进需求）。
 */
public interface ConjunctPushdown {
    ConnectorExpression originalConjunct();           // 原始 conjunct
    Optional<ConnectorExpression> pushedExpression();  // 下推到源的部分（EXACT 有值，IN_MEMORY 为空）
    ConnectorExpression residualExpression();          // 残余：引擎在上方补 Filter
    PushdownMode mode();                               // 此 conjunct 的模式
}

/**
 * 谓词下推完整结果。
 * v2: 从单一 mode 改为 List<ConjunctPushdown>，支持混合语义。
 * v2.1: 改为 interface，新字段通过 default 方法添加。
 */
public interface FilterResult {
    TableHandle pushedTable();                         // 烤入已下推 EXACT conjunct 的新 handle
    List<ConjunctPushdown> conjunctResults();          // 逐 conjunct 结果（含残余）

    /**
     * 引擎需在上方执行的残余谓词。
     * = 所有 conjunct 的 residual 的 AND。
     * v2.1 修复: 使用正确的 Stream.reduce(BinaryOperator) 重载。
     */
    default ConnectorExpression combinedResidual() {
        return conjunctResults().stream()
            .map(ConjunctPushdown::residualExpression)
            .reduce(ConnectorExpression::logicalAnd)
            .orElse(ConnectorExpression.TRUE);
    }

    /** v2.1 预留：下推后的统计估计（供 cost 模型后续优化） */
    default Optional<PlanStats> estimatedStats() { return Optional.empty(); }
}

// ====== 聚合下推结果（v2: 中间状态 + 合并函数 + HAVING 类型修复）======

public interface AggregateResult {
    TableHandle pushedTable();
    AggregateMode mode();                               // COMPLETE / PARTIAL
    List<IntermediateAggregate> intermediateAggregates(); // v2 新增：中间状态 + 合并函数
    ConnectorExpression remainingHaving();              // v2 修复：从 TupleDomain 改为 ConnectorExpression
    List<AggregateFunction> remainingAggregates();      // 不可下推的聚合函数
}

public enum AggregateMode {
    NONE,       // 不下推
    PARTIAL,    // 源做部分聚合，引擎做最终合并（两阶段）
    COMPLETE    // 源做完整聚合，引擎直接输出
}

/**
 * v2 新增：部分聚合的中间状态契约（修复 Top5 #4）。
 * 每个下推的聚合函数必须声明其中间类型和合并函数。
 * v2.1: 改为 interface。
 */
public interface IntermediateAggregate {
    FunctionSignature originalAggregate();             // 原始聚合（如 AVG）
    Type intermediateType();                           // 中间状态类型（如 AVG → ROW(sum, count)）
    MergeFunction mergeFunction();                     // 合并函数
    String intermediateColumnAlias();                  // 中间结果列别名
}

/**
 * v2 新增：合并函数契约。
 * 引擎用此合并来自不同分区/batch 的部分聚合结果。
 * v2.1 修复: 加 mergeAll 默认方法（修复 §16.1 测试中引用的未定义方法）。
 */
public interface MergeFunction {
    /** 合并两个部分聚合状态 */
    Object merge(Object partial1, Object partial2);

    /** v2.1 新增：合并多个部分聚合状态（默认基于 merge 逐个合并） */
    default Object mergeAll(List<Object> partials) {
        if (partials.isEmpty()) {
            throw new IllegalArgumentException("Cannot merge empty partial list");
        }
        Object result = partials.get(0);
        for (int i = 1; i < partials.size(); i++) {
            result = merge(result, partials.get(i));
        }
        return result;
    }

    /** 序列化中间状态（用于跨网络传输） */
    byte[] serialize(Object intermediateState);

    /** 反序列化中间状态 */
    Object deserialize(byte[] data);

    /** 从中间状态计算最终结果 */
    Object finalize(Object intermediateState);
}

// 常见聚合的合并函数示例：
// SUM:   intermediate=BigDecimal, merge=(a,b)->a+b, finalize=identity
// COUNT: intermediate=long, merge=(a,b)->a+b, finalize=identity
// AVG:   intermediate=Row(sum,count), merge=Row(a.sum+b.sum, a.count+b.count), finalize=sum/count
// MIN:   intermediate=同类型, merge=(a,b)->min(a,b), finalize=identity
// MAX:   intermediate=同类型, merge=(a,b)->max(a,b), finalize=identity

// ====== Join 下推结果 ======

public interface JoinResult {
    Optional<TableHandle> pushedJoinTable();            // 同源时：已下推 join 的新 handle
    Optional<TableHandle> pushedLeft();                 // 跨源时：左表（可能已加过滤）
    Optional<TableHandle> pushedRight();                // 跨源时：右表（可能已加过滤）
    ConnectorExpression remainingCondition();           // v2: 改为 ConnectorExpression

public enum CrossSourceStrategy {
    FULL_PUSH,           // 同源，整 join 下推
    FILTER_EACH_SIDE,    // 各自下推过滤，引擎本地 join

    SEMI_JOIN            // 半连接下推
}

// ====== TopN 下推结果（v2: collation 证明 + 默认安全）======

public interface TopNResult {
    TableHandle pushedTable();
    boolean orderGuaranteed();
    boolean limitGuaranteed();
```

### 3.6 两层接入设计

### 4.0 PushedPlan 类型定义（v2.1 新增，修复 F3）

```java
/**
 * v2.1 新增：下推计划，deparse 层的输入。
 * 包含规划器选中路径的全部下推信息，deparse 据此生成一条远程 SELECT。

    /** 下推的投影列表（SELECT 列表），空 = SELECT * */

    Optional<List<ColumnHandle>> groupingKeys();

    /** 已下推的 HAVING 谓词（仅可运部分），空 = 无 HAVING 或 HAVING 未下推 */
    Optional<ConnectorExpression> pushedHaving();

    Optional<List<SortItem>> sortItems();

    /** LIMIT 值，空 = 无 LIMIT */
    Optional<Long> limit();

    /** 游标每批拉取行数（默认 100） */
    default int fetchSize() { return 100; }

    /** 残余谓词（引擎在上方执行的部分） */
    default ConnectorExpression residual() {
        return conjunctResults().stream()
            .map(ConjunctPushdown::residualExpression)
            .reduce(ConnectorExpression::logicalAnd)
            .orElse(ConnectorExpression.TRUE);
    }
}
```

### 4.2 deparse 算法

```java
public class DefaultSqlDeparser implements SqlDeparser {

    public DeparsedQuery deparseSelectStmt(PushedPlan plan, SqlDialect dialect) {
        StringBuilder sql = new StringBuilder("SELECT ");
        List<Object> params = new ArrayList<>();
        List<Integer> retrievedAttrs = new ArrayList<>();

        // 1. SELECT 列表（投影下推的部分）
        deparseTargetList(sql, retrievedAttrs, plan.projections(), dialect, params);

        // 2. FROM 子句
        sql.append(" FROM ");
        deparseFromClause(sql, plan.tableHandle(), dialect);

        // 3. WHERE（仅 EXACT conjunct 翻译为 SQL）
        List<ConnectorExpression> whereConjuncts = plan.conjunctResults().stream()
            .filter(cp -> cp.mode() == PushdownMode.EXACT)
            .map(cp -> cp.pushedExpression().orElseThrow())
            .toList();
        if (!whereConjuncts.isEmpty()) {
            sql.append(" WHERE ");
            appendConditions(sql, whereConjuncts, dialect, params);
        }

        // 4. GROUP BY / HAVING
        if (plan.hasAggregate()) {
            sql.append(" GROUP BY ");
            deparseGroupBy(sql, plan.groupingKeys(), dialect);
            if (plan.hasHaving()) {
                // 仅可运的 HAVING 部分翻译
                sql.append(" HAVING ");
                appendHaving(sql, plan.pushedHaving(), dialect, params);
            }
        }

        // 5. ORDER BY
        if (plan.hasSort()) {
            sql.append(" ORDER BY ");
            deparseOrderBy(sql, plan.sortItems(), dialect);
        }

        // 6. LIMIT / OFFSET
        if (plan.hasLimit()) {
            sql.append(" LIMIT ").append(plan.limit());
        }

        return new DeparsedQuery(sql.toString(), retrievedAttrs, params, plan.fetchSize());
    }

    /**
     * 表达式 deparse：递归遍历 ConnectorExpression 树。
     */
    private void deparseExpr(StringBuilder sql, ConnectorExpression expr,
                             SqlDialect dialect, List<Object> params) {
        switch (expr) {
            case Variable v -> sql.append(dialect.quoteIdentifier(v.column().name()));
            case Constant c -> {
                if (shouldParameterize(c)) {
                    params.add(c.value());
                    sql.append("$").append(params.size());
                } else {
                    sql.append(dialect.formatLiteral(c.value(), c.type()));
                }
            }
            case Call call -> {
                sql.append(call.function().name()).append("(");
                for (int i = 0; i < call.args().size(); i++) {
                    if (i > 0) sql.append(", ");
                    deparseExpr(sql, call.args().get(i), dialect, params);
                }
                sql.append(")");
            }
            case Comparison cmp -> {
                deparseExpr(sql, cmp.left(), dialect, params);
                sql.append(" ").append(cmp.op().symbol()).append(" ");
                deparseExpr(sql, cmp.right(), dialect, params);
            }
            case Logical log -> {
                String sep = log.op() == LogicalOperator.AND ? " AND " : " OR ";
                for (int i = 0; i < log.terms().size(); i++) {
                    if (i > 0) sql.append(sep);
                    sql.append("(");
                    deparseExpr(sql, log.terms().get(i), dialect, params);
                    sql.append(")");
                }
            }
            case Cast cast -> {
                sql.append("CAST(");
                deparseExpr(sql, cast.expr(), dialect, params);
                sql.append(" AS ").append(dialect.typeName(cast.targetType())).append(")");
            }
            case Special s -> deparseSpecial(sql, s, dialect, params);
            case TupleDomain td -> deparseTupleDomain(sql, td, dialect, params);
        }
    }
}
```

### 5.3 静态 + 动态组合（v2: 模式感知）

```java
/**
 * 同一扫描可同时有静态和动态下推。
 * v2 修复：组合时区分 EXACT 和 CONSERVATIVE 静态谓词。
 */
public record ScanContext(
    TableHandle table,
    List<ConjunctPushdown> staticConjuncts,           // v2: 逐 conjunct（含模式）
    DynamicFilter dynamicFilter,
    Set<ColumnHandle> dynamicFilterColumns
) {
    /**
     * v2 修复：模式感知的残余组合。
     * 
     * - EXACT 静态 conjunct：源已执行，残余仅含不可翻译的部分
     * - CONSERVATIVE 静态 conjunct：残余 = 完整原始 conjunct（每行仍需检查）
     * - 动态谓词：到达后与 CONSERVATIVE 残余取 AND
     * 
     * 引擎在上方执行的残余 Filter =
     *   (所有 CONSERVATIVE + IN_MEMORY 静态残余) ∧ (动态谓词)
     *   （EXACT 静态残余也包含，但通常为空或不可翻译部分）
     */
    public ConnectorExpression computeEngineResidual(ConnectorExpression dynamicPredicate) {
        // 静态残余：所有非 EXACT conjunct 的完整原始谓词 + EXACT 的残余
        // v2.1 修复: 使用正确的 Stream.reduce(BinaryOperator) 重载
        ConnectorExpression staticResidual = staticConjuncts.stream()
            .map(ConjunctPushdown::residualExpression)
            .reduce(ConnectorExpression::logicalAnd)
            .orElse(ConnectorExpression.TRUE);

        // 动态谓词到达时，与静态残余取 AND
        if (dynamicPredicate != null) {
            return ConnectorExpression.logicalAnd(staticResidual, dynamicPredicate);
        }
        return staticResidual;
    }

    /**
     * v2 新增：动态谓词到达时，是否需要重新做数据跳过。
     * 仅当存在 CONSERVATIVE 静态 conjunct 或源支持动态数据跳过时才需要。
     */
    public boolean shouldReSkipOnDynamic() {
        return staticConjuncts.stream().anyMatch(cp -> cp.mode() == PushdownMode.CONSERVATIVE);
    }
}
```

## 六、下推决策引擎

### 6.1 算子可下推判定矩阵

| 算子            | 可下推条件                               | 残余                                 | 保证标志                                              |
| ------------- | ----------------------------------- | ---------------------------------- | ------------------------------------------------- |
| **Filter**    | 谓词可运（shippable）                     | 不可运的 conjunct（逐 conjunct 模式）       | 逐 conjunct EXACT/CONSERVATIVE/IN_MEMORY           |
| **Project**   | 所有表达式可运                             | 无（全推或全不推）                          | -                                                 |
| **Aggregate** | 所有 agg 函数可运 + 分组列在源 + HAVING 可运     | 不可运的 HAVING（`ConnectorExpression`） | mode (COMPLETE/PARTIAL)                           |
| **Join**      | 两侧同源 + join 条件可运 + join 类型安全        | 不可运的 join 条件                       | -                                                 |
| **Sort**      | 源支持 ORDER BY + 排序键可运 + collation 兼容 | 无                                  | orderGuaranteed + sortCollation                   |
| **Limit**     | 源支持 LIMIT                           | 无                                  | limitGuaranteed（默认 false）                         |
| **TopN**      | Sort+Limit 组合下推                     | 无                                  | orderGuaranteed + sortCollation + limitGuaranteed |
| **Window**    | 通常**不可下推**（帧语义复杂）                   | 全部                                 | -                                                 |
| **Union**     | 每分支独立下推                             | 无                                  | -                                                 |

**Join 类型安全性**：

- INNER / LEFT / RIGHT / FULL OUTER → 可推（条件可运时）
- SEMI / ANTI → 需转为 IN/EXISTS 子查询（部分源支持）
- CROSS → 可推但无过滤收益

**聚合下推陷阱**：

- 分组基数 ≈ 输入行数（如按 rowid 分组）→ 部分聚合不缩减数据，反而增加开销
- `COUNT(DISTINCT x)` → 部分结果不可合并（集合需 union 非求和），只允许 complete 下推
- 框架通过 `IntermediateAggregate.mergeFunction` 校验可合并性；不可合并的聚合（如 `ARRAY_AGG`）仅允许 COMPLETE
- `estimatedOutputRows()` 让规划器在分组基数陷阱时拒绝下推

### 6.2 分解算法：枚举候选路径 + Memo + 剪枝 + 候选上限（v2 修订）

```java
/**
 * 下推路径构建器（v2 修订：加 memo + 剪枝 + 候选上限，修复路径爆炸）
 *
 * v2 关键改进：
 * 1. 先定 join 顺序，再逐节点枚举下推候选（解耦 join 排序与下推）
 * 2. Memo 缓存每个 RelSubset 的最优下推路径（参考 Calcite Volcano）
 * 3. 剪枝：候选路径下界 cost > 当前最优 → 立即剪除
 * 4. 候选上限：每节点最多 3-5 条候选（可配置）
 */
public class PushdownPathBuilder {

    private final Memo memo = new Memo();           // RelSubset → 最优路径缓存
    private final int maxCandidatesPerNode = 5;      // 每节点候选上限
    private final PushdownCostModel costModel;

    public List<PlanPath> buildCandidates(LogicPlan plan, ConnectorSession session) {
        List<PlanPath> paths = new ArrayList<>();

        // 路径1：LOCAL_ONLY（兜底）
        paths.add(PlanPath.localOnly(plan));

        // 用 isPushable 纯检查（无副作用），而非 applyXxx
        if (connector.isFilterPushable(session, table, predicate)) {
            // 路径2：PUSH_FILTER（仅构建路径描述，不执行 applyFilter）
            PushdownCost estimatedCost = costModel.estimatePush(plan, predicate, stats);

            // 剪枝：如果下界 cost 已劣于当前最优，跳过
            if (estimatedCost.lowerBound() < memo.getBestCost(plan)) {
                paths.add(PlanPath.pushFilter(plan, estimatedCost));
            }
        }

        // 路径3-6：类似检查 isProjectionPushable / isAggregatePushable / ...

        // 候选上限：最多保留 maxCandidatesPerNode 条
        return paths.stream()
            .sorted(Comparator.comparingDouble(p -> p.cost().lowerBound()))
            .limit(maxCandidatesPerNode)
            .toList();
    }

    /**
     * 对选中的路径执行 applyXxx（仅一次，修复 Top5 #3）。
     */
    public PlanNode executeChosenPath(PlanPath chosen, ConnectorSession session) {
        // 仅对选中路径调用 applyFilter（有副作用，生成新 TableHandle）
        FilterResult result = connector.applyFilter(session, table, predicate, snapshot);

        // 校验残余不变量（debug/test）
        invariantValidator.validateFilter(predicate, result.conjunctResults());

        return buildPhysicalPlan(result);
    }
}
```

### 6.3 下推与 join 排序的解耦

```
1. 先做 join 排序（标准 join enumeration，不虑下推）
2. 对排好序的 join 树，逐节点枚举下推候选
3. 候选仅限当前节点（不跨节点组合，避免指数爆炸）
4. Memo 缓存每个 RelSubset 的最优下推路径
```

> **v2 关键设计**：下推与 join 排序解耦——先定 join 顺序，再逐节点枚举下推。这避免了 join 顺序 × 下推候选的组合爆炸。

## 七、数据跳过下推

### 7.1 三级跳过流水线

```
谓词 → ┌─────────────────────────────────────────────────┐
       │ 1. 分区裁剪 (Partition Prune)                    │  用分区值
       │    输入: 谓词的分区投影（InclusiveProjection）     │
       │    输出: 候选文件列表                             │
       ├─────────────────────────────────────────────────┤
       │ 2. 粗粒度裁剪 (Manifest/Stats Prune)             │  用文件级 min/max 统计
       │    输入: 谓词的统计投影（InclusiveProjection）     │
       │    输出: 精简文件列表                             │
       ├─────────────────────────────────────────────────┤
       │ 3. 细粒度裁剪 (Row-Group/Page Prune)             │  Parquet reader 内部
```

### 8.1 Volatility 分类

### 8.2 ShippabilityChecker（v2: 逐源白名单 + 快照钉入）

```java
public class ShippabilityChecker {

    private final ShippabilityCache cache;

    /**
     * v2 修复：STABLE 函数需要快照钉入。
     * now()/current_date 不能直接下推——必须替换为快照时间戳常量。
     */
    public boolean isShippable(ConnectorExpression expr, ConnectorSession session,
                                SnapshotContext snapshot) {
        return switch (expr) {
            case Variable v -> isInSource(v) && typeCompatible(v, session);

            case Constant c -> true;

            case Call f -> {
                // v2 修复：白名单逐源（非全局）
                // v2.1 修复：通过 f.function() 访问 FunctionSignature，再取 volatility
                FunctionSignature sig = f.function();
                boolean builtinOk = cache.isBuiltinForSource(sig, session.serverId());
                boolean extOk = cache.isWhitelistedExtension(sig, session.serverId());
                boolean argsOk = f.args().stream()
                    .allMatch(a -> isShippable(a, session, snapshot));
                boolean volatilityOk = sig.volatility() != FunctionVolatility.VOLATILE;

                // v2 新增：STABLE 函数需要快照钉入
                boolean snapshotOk = true;
                if (sig.volatility() == FunctionVolatility.STABLE) {
                    snapshotOk = snapshot != null 
                        && sourceSupportsPinnedTimestamp(session.serverId());
                    // 推到源的 STABLE 函数值会被替换为快照时间戳常量
                    // 如 now() → {pinned_timestamp}
                }

                yield builtinOk && extOk && argsOk && volatilityOk && snapshotOk
                    && collationCompatible(sig, session)
                    && timezoneCompatible(sig, session)
                    && semanticCompatible(sig, session);  // v2: 语义兼容(除法/varchar/decimal)
            }

            default -> false;  // 安全默认：不认识的类型不推
        };
    }
}
```

### 9.2 判定公式

```java
    public boolean shouldPush(PushdownCandidate c) {
        long N = c.baseCardinality();
        double sel = c.selectivity();
        long pushedRows = (long)(N * sel);

        double pushCost = sourceStartupCost
                       + remoteCompute(N, c.predicateComplexity())
                       + transferCost(pushedRows, c.rowSize());

        double noPushCost = transferCost(N, c.rowSize())
                          + localCompute(N, c.predicateComplexity());

        return pushCost < noPushCost;
    }
```

```java
/**
 * v2 新增：回退估计器。
 * 当连接器不提供统计时，使用启发式估计选择率。
 */
public class FallbackEstimator {

    /**
     * 估计谓词选择率。
     * 
     * 启发式规则：
     * - 等值 + 唯一/PK 列 → sel ≈ 1/rowCount
     * - 等值 + 非唯一列 → sel ≈ 1/NDV（已知）或默认 0.1
     * - 范围谓词 → sel from min/max 区间重叠，或默认 0.33
     * - LIKE 前缀 → sel ≈ 0.1
     * - IS NULL → sel ≈ nullCount/rowCount（已知）或默认 0.05
     * - 多列 → 假设独立，选择率相乘
     */
    public double estimateSelectivity(ConnectorExpression predicate,
                                       Optional<TableStatistics> stats) {
        if (stats.isEmpty()) {
            return estimateWithoutStats(predicate);  // 纯启发式
        }
        return estimateWithStats(predicate, stats.get());
    }

    private double estimateWithoutStats(ConnectorExpression predicate) {
        return switch (predicate) {
            case Comparison c when c.op() == Operator.EQ -> 0.005;   // 等值默认
            case Comparison c when c.op().isRange() -> 0.33;          // 范围默认
            case Special s when s.kind() == SpecialKind.IS_NULL -> 0.05;
            case Special s when s.kind() == SpecialKind.LIKE -> 0.1;
            case Logical l when l.op() == LogicalOperator.AND ->
                l.terms().stream()
                    .mapToDouble(t -> estimateWithoutStats(t))
                    .reduce(1.0, (a, b) -> a * b);  // 独立假设
            case Logical l when l.op() == LogicalOperator.OR ->
                1.0 - l.terms().stream()
                    .mapToDouble(t -> 1.0 - estimateWithoutStats(t))
                    .reduce(1.0, (a, b) -> a * b);  // 独立假设
            default -> 0.1;  // 未知谓词默认
        };
    }
}
```

## 十一、跨源 Join 策略

当 Join 跨两个不同源，不能直接下推整个 Join：

| 策略               | 适用场景     | 机制                                   |
| ---------------- | -------- | ------------------------------------ |
| **独立下推**         | 两侧无关联    | 各自下推谓词，引擎本地 Join                     |
| **广播下推**         | 小侧可广播    | 广播小侧 join key → 大侧用 `IN (keys)` 动态下推 |
| **Semi-join 下推** | 只需大侧缩减   | 提取小侧 distinct key，下推到大侧作过滤           |
| **动态分区裁剪**       | 分区表 Join | 广播小侧 → 运行时裁剪大侧分区（Spark DPP）          |
| **Bloom filter** | 大基数 Join | 小侧建 Bloom filter → 广播 → 大侧逐行/块跳过     |

```java
public enum JoinPushdownStrategy {
    FULL_JOIN,        // 同源：整个 Join 下推
    FILTER_ONLY,      // 跨源：各自下推谓词，引擎 Join
    BROADCAST,        // 跨源：广播小侧 key，大侧 IN-list 动态下推
    SEMI,             // 跨源：小侧 distinct key 下推大侧
    BLOOM_FILTER      // 跨源：Bloom filter 运行时过滤
}
```

### Spark 动态分区裁剪（DPP）

- Fact 表分区列 `k` 与 Dimension 表 join on `k`
- Spark 插入 `SubqueryBroadcast` 收集 Dimension 的 distinct `k`
- 包装为 `DynamicPruningExpression(InSet(k, values))` 附到 Fact 扫描

### Spark Bloom Filter Runtime（3.3+）

- Build 侧通过 `BloomFilterAggregate` 聚合值到 Bloom filter
- Probe 侧注入 `BloomFilterMightContain(bf, probeCol)` 作为扫描过滤

### 12.1 运行时错误回退策略

```java
/**
 * 下推执行失败时的回退策略。
 */
public class PushdownFallbackHandler {

    private final int maxRetries = 3;
    private final Duration retryDelay = Duration.ofMillis(100);
    private final CircuitBreaker circuitBreaker;

    /**
     * 执行下推查询，失败时自动回退。
     * 
     * 回退链：
     * 1. 重试（带退避，最多 maxRetries 次）
     * 2. 重试耗尽 → 回退到本地执行：fallbackToFullScan() + 引擎内存过滤
     * 3. 连续失败 → 熔断：临时禁用该源的下推（冷却期）
     */
    public ConnectorPageSource executeWithFallback(PushedPlan plan, ConnectorSession session) {
        try {
            return retryExecute(plan, session, maxRetries, retryDelay);
        } catch (SourceUnavailableException e) {
            // 回退到全扫 + 本地过滤
            TableHandle original = connector.fallbackToFullScan(plan.tableHandle());
            return fullScanWithLocalFilter(original, plan.residual(), session);
        }
    }

    /**
     * 熔断器：连续失败时临时禁用下推。
     */
    public boolean isPushdownEnabled(String serverId) {
        return circuitBreaker.isClosed(serverId);
        // 熔断打开时，该源的查询全部走本地执行
        // 冷却期过后半开试探
    }
}
```

### 13.2 STABLE 函数快照钉入

```java
/**
 * STABLE 函数（now/current_date）下推时，必须替换为快照时间戳常量。
 * 不能让源在执行时重新评估 now()——会破坏快照一致性。
 * 
 * 如：WHERE ts <= now() 
 *   → WHERE ts <= {pinned_timestamp}  （pinned_timestamp = queryTimestamp）
 */
public class StableFunctionPinner {

    public ConnectorExpression pinStableFunctions(ConnectorExpression expr,
                                                    SnapshotContext snapshot) {
        // 遍历表达式树，将 STABLE 函数调用替换为快照时间戳常量
        return transform(expr, node -> {
            if (node instanceof Call call && call.volatility() == FunctionVolatility.STABLE) {
                Object pinnedValue = evaluateAt(call, snapshot.queryTimestamp());
                return new Constant(pinnedValue, call.type());
            }
            return node;
        });
    }
}
```

> 
> - 单源内：快照隔离（源原生支持时）
> - 跨源：每个源独立快照，时间戳对齐但非原子
> - STABLE 函数：钉入查询快照时间戳，保证一致性

## 十四、安全与访问控制（v2 新增）

v1 缺失安全/RLS 模型——下推是否绕过引擎行级安全未定义。v2 补全。

### 14.1 行级安全（RLS）

```java
/**
 * 下推不绕过引擎 RLS。
 * 引擎 RLS 谓词作为 ADDITIONAL conjunct 追加到下推谓词中。
 * 
 * 最终下推到源的谓词 = 用户查询谓词 ∧ 引擎 RLS 谓词
 * 
 * 如果源有自己的 RLS，两者都生效（更严格者赢）。
 */
public class RlsAwarePushdown {

    public ConnectorExpression appendRlsPredicates(ConnectorExpression userPredicate,
                                                     ConnectorSession session,
                                                     TableHandle table) {
        // 获取引擎 RLS 谓词
        List<ConnectorExpression> rlsPredicates = rlsService.getRlsPredicates(
            session.user(), table);

        // 合并：用户谓词 ∧ RLS 谓词
        ConnectorExpression combined = userPredicate;
        for (ConnectorExpression rls : rlsPredicates) {
            combined = ConnectorExpression.logical(LogicalOperator.AND, combined, rls);
        }
        return combined;
    }
}
```

### 14.2 列脱敏

```java
/**
 * 脱敏列不下推。
 * 引擎在 fetch 后应用脱敏函数。
 */
public class MaskingAwarePushdown {

    public PushdownDecision checkMasking(TableHandle table, List<ColumnHandle> projectedColumns,
                                          ConnectorSession session) {
        Set<ColumnHandle> maskedColumns = maskingService.getMaskedColumns(
            session.user(), table);

        if (projectedColumns.stream().anyMatch(maskedColumns::contains)) {
            // 有脱敏列：可以下推谓词，但不能下推脱敏列的投影
            // 引擎 fetch 原始值后应用脱敏函数
            return PushdownDecision.pushFilterOnly(maskedColumns);
        }
        return PushdownDecision.pushAll();
    }
}
```

## 十五、子查询与 CTE 处理（v2 新增）

v1 完全未涉及子查询/CTE——真实查询中必有。v2 补全。

### 15.1 处理策略

| 子查询类型                      | 处理策略                                    | 下推行为       |
| -------------------------- | --------------------------------------- | ---------- |
| **标量子查询**（SELECT/WHERE）    | 先去相关（Calcite 风格）                        | 去相关后的形式可下推 |
| **相关子查询**                  | 相关列在同源 → 推为相关子查询或 JOIN；跨源 → 引擎去相关为 JOIN | 同源可推，跨源去相关 |
| **CTE**                    | 默认内联（每个引用独立下推）；物化 CTE 留引擎               | 内联可推，物化不可推 |
| **EXISTS/IN**              | 源支持子查询 → 推；否则改写为 semi-join              | 源支持时可推     |
| **UNION/INTERSECT/EXCEPT** | 每分支独立下推                                 | 分支级下推      |

### 15.2 去相关顺序

```
查询 → 解析 → 去相关（子查询→JOIN）→ 类型检查 → join 排序 → 下推分析
```

> **关键**：去相关是下推的**前置步骤**。相关子查询在去相关后才能判断是否可下推。引擎应使用 Calcite 风格的去相关（`Decorrelator`）将相关子查询转为 JOIN，然后对 JOIN 树做下推分析。

```java
/**
 * 基于属性的测试：随机谓词 → 验证下推 + 残余 = 本地执行结果。

@Property(tries = 10000)
void testPushdownCorrectness(
    @ForAll("randomPredicate") ConnectorExpression predicate,
    @ForAll("randomTable") TableHandle table

    assertEquals(toSet(localRows), toSet(finalRows));
}
```

## 二十、反模式警示

| 反模式                           | 后果                        | 正确做法                                  |
| ----------------------------- | ------------------------- | ------------------------------------- |
| 把 CONSERVATIVE 当 EXACT        | 漏行（静默错误）                  | 逐 conjunct 模式；残余不变量校验                 |
| 用 TupleDomain 表达所有谓词          | 函数/算术谓词无法下推               | 用 ConnectorExpression IR              |
| 贪心切割                          | 错过更优方案                    | 枚举候选 + Memo + 剪枝                      |
| applyFilter 在路径构建时执行          | 浪费工作                      | isPushable 纯检查 + apply 仅选中路径          |
| 为未选中路径生成 SQL                  | 浪费规划时间                    | deparse 仅对选中路径                        |
| 全局缓存 shippability             | collation/timezone 变化导致错误 | 每查询重新校验                               |
| BUILTIN 白名单全局共享               | 函数语义跨源不同（除法/varchar）      | 逐源白名单 + 语义兼容检查                        |
| STABLE 函数直接下推                 | 快照不一致                     | 钉入快照时间戳常量                             |
| 动态过滤无内存上限                     | build 侧 OOM               | maxDistinctValues + 降级 Bloom filter   |
| 动态过滤无 ERROR 状态                | build 失败时 probe 挂死        | ERROR 终态 + 回退全扫                       |
| guarantee flag 无 collation 证明 | 排序 collation 不匹配→错误结果     | 携带 sortCollation + 默认安全(false)        |
| 部分聚合无合并契约                     | 不可合并函数结果错误                | IntermediateAggregate + MergeFunction |
| remainingHaving 用 TupleDomain | HAVING 含聚合表达式，类型错误        | 用 ConnectorExpression                 |
| 无统计来源时声称"成本驱动"                | 退化为猜测                     | FallbackEstimator + EXPLAIN 标注        |
| 下推绕过引擎 RLS                    | 安全漏洞                      | RLS 谓词追加到下推谓词                         |
| 脱敏列下推到源                       | 脱敏失效                      | 脱敏列不下推，引擎 fetch 后脱敏                   |
| 无错误回退                         | 源失败时查询报错                  | 重试 + fallbackToFullScan + 熔断          |
| 无一致性测试                        | 连接器 bug 导致静默错误            | 强制通过一致性测试套件                           |
| 部分聚合分组基数 ≈ 行数                 | 增加远程计算不减少传输               | cost 模型判定，基数高时不推                      |
| join 排序与下推耦合                  | 组合爆炸                      | 解耦：先排序再逐节点下推                          |
